Шифрование токенов

Все чувствительные данные зашифрованы и недоступны в открытом виде

  • AES-256-GCM — токены ботов (BotFather) и Leadteh API-токены шифруются алгоритмом AES-256-GCM перед сохранением в базу данных. Это тот же стандарт, который используют банки и госструктуры.
  • Уникальный вектор инициализации (IV) — для каждого токена генерируется случайный IV, что делает невозможным определение одинаковых токенов по зашифрованным данным.
  • AuthTag — каждый зашифрованный блок содержит тег аутентичности, защищающий от подмены данных в базе.
  • Ключ шифрования хранится только в файле .env на сервере и нигде не передается. Без ключа расшифровать данные невозможно.
  • Маскирование — в API и интерфейсе токены ботов отображаются в маскированном виде (1234****:****DEF). Полный токен не передается клиенту.

Авторизация через Telegram

Многоуровневая проверка личности без паролей

  • HMAC-SHA-256 — каждый запрос на вход проверяется криптографической подписью Telegram. Подделать запрос авторизации невозможно — подпись формируется серверами Telegram.
  • Защита от replay-атак — подпись действительна только 24 часа. Повторное использование старых данных авторизации будет отклонено.
  • Timing-safe сравнение — проверка подписи выполняется через timingSafeEqual, что исключает атаки по времени отклика (timing attacks).
  • Сессии — после успешной авторизации создается Bearer-токен (32 случайных байта через crypto.randomBytes). Сессия действует 24 часа.
  • Без паролей — в системе нет паролей, которые можно украсть или подобрать. Вход только через Telegram.

Изоляция данных между аккаунтами

Один пользователь никогда не увидит данные другого

  • Tenant ID из сессии — идентификатор аккаунта берется из серверной сессии, а не из запроса клиента. Подменить его невозможно.
  • Каждый SQL-запрос фильтрует данные по tenant_id. Рассылки, боты, настройки, диалоги — всё привязано к конкретному аккаунту.
  • Загрузки файлов изолированы по папкам: каждый аккаунт хранит медиа в отдельной директории. Доступ к чужим файлам заблокирован.
  • Защита от IDOR — даже зная ID чужого бота или рассылки, получить к ним доступ невозможно — сервер всегда проверяет принадлежность ресурса.

Защита от атак

Многослойная защита от распространенных веб-угроз

  • SQL Injection — все запросы к базе данных параметризованы. Пользовательские данные никогда не подставляются в SQL напрямую.
  • XSS (межсайтовый скриптинг) — все пользовательские данные (имена, теги, тексты сообщений) экранируются через escapeHtml() и escapeAttr() перед отображением.
  • Path Traversal — при работе с файлами проверяется, что путь не выходит за пределы разрешенной директории. Символы ../ блокируются.
  • CORS — запросы к API разрешены только с доверенных доменов (Telegram WebApp, leadtehsms.ru). Запросы с неизвестных сайтов блокируются.
  • Rate Limiting — ограничение частоты запросов: 15 попыток авторизации/мин, 60 API-запросов/мин, 3 загрузки файлов/мин. Защищает от brute-force и DDoS.
  • Лимит размера запросов — API принимает максимум 2 МБ данных. Увеличенный лимит (30 МБ) только для загрузки медиа. Защита от исчерпания памяти сервера.

HTTP-заголовки безопасности

Дополнительные инструкции для браузера

  • X-Content-Type-Options: nosniff — браузер не будет угадывать тип файла, предотвращая атаки через подмену MIME-типа.
  • X-Frame-Options: DENY — страницу нельзя встроить в iframe на стороннем сайте (защита от clickjacking).
  • Referrer-Policy — при переходе на внешние сайты не передается полный URL страницы.
  • Permissions-Policy — заблокирован доступ к камере, микрофону и геолокации.
  • HTTPS — весь трафик шифруется SSL-сертификатом (Let's Encrypt). Перехватить данные между вами и сервером невозможно.

Защита Webhook

Только настоящие уведомления от Telegram

  • Secret Token — при регистрации Webhook устанавливается секретный токен. Каждый входящий запрос проверяется — поддельные уведомления отклоняются.
  • Боты тенантов не перехватываются — платформа использует токены ботов только для отправки сообщений. Webhook бота остается за Leadteh или другим сервисом. Мы не вмешиваемся в логику вашего бота.
🗃

Хранение данных

Где хранятся данные и кто имеет к ним доступ

  • База данных хранится на выделенном VPS-сервере в защищенной директории. Доступ только у администратора платформы.
  • Токены зашифрованы — даже при физическом доступе к базе данных токены невозможно прочитать без ключа шифрования.
  • Автосохранение — данные сохраняются на диск каждые 5 секунд и при каждом изменении (с debounce 500 мс). При выключении сервера база корректно сохраняется (graceful shutdown).
  • Нет данных в браузере — токены авторизации хранятся только в оперативной памяти приложения, а не в localStorage. При закрытии Mini App все данные стираются.
  • API никогда не возвращает полные токены ботов или API-ключи в ответах. Суперадмин видит только факт наличия токена, но не его значение.

Генерация идентификаторов

Криптографически безопасные и непредсказуемые

  • crypto.randomUUID() — все идентификаторы рассылок генерируются криптографически безопасным генератором случайных чисел. Угадать или перебрать ID невозможно.
  • crypto.randomBytes() — сессионные токены, ключи шифрования и секреты webhook генерируются через криптографический генератор операционной системы.
